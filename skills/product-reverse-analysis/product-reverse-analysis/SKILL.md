# 产品逆向分析 (Product Reverse Analysis)

本 skill 的目标：当用户提供一张产品界面截图时，像资深产品经理 + 首席设计师 + 架构师三位一体的顾问那样，从 UI 表象逆向推导产品背后的业务闭环、技术实现与设计哲学。

## 工作流程

1. **确认输入**：确认用户已提供截图。如果没有截图，先礼貌请求用户上传一张清晰的产品界面截图。
2. **观察与读图**：先在心里走一遍界面：有哪些区域？哪些元素权重最高？颜色/字体/留白告诉了你什么？信息密度如何？
3. **按"输出结构"生成完整报告**：严格使用下方的 Markdown 模板，不省略任何一级小节。
4. **保持洞察力**：避免描述性的流水账（"顶部有一个按钮"），要输出推断性的结论（"顶部按钮的位置+配色暗示这是高频核心转化入口，团队对该路径有强埋点监控需求"）。

## 为什么要这样做

- **逆向分析的价值不在于"看到了什么"，而在于"为什么这么做"**。一张截图背后藏着 PRD、用研报告、AB 测试结果、合规约束、技术栈选型。报告必须把这些隐性信息挖出来，才对读者有价值。
- **结构化输出**便于用户在后续工作（竞品分析、设计评审、架构讨论）中直接复用，所以 Markdown 模板必须严格遵守，不得自由发挥章节顺序。
- **Mermaid 流程图**强制存在，是为了让业务逻辑从文字推演上升为可视化闭环，读者一眼能看懂主路径。

## 输出结构（严格遵守）

语言：中文。语气：专业、严谨、具有洞察力。格式：Markdown，严格按下列模板。

```markdown

- **产品定位**：用一句话定义该产品/界面的核心价值。
- **目标用户**：分析该界面服务于哪类特定人群（年龄、职业、使用阶段、心智模型）。
- **核心场景**：描述用户在什么样的情境下会打开这个界面，解决什么痛点。

## Step 2: 模块拆解

### 2.1 模块概览表

| 模块名称 | 视觉位置 | 核心作用 | 优先级 (P0/P1/P2) |
| :--- | :--- | :--- | :--- |

### 2.2 深度模块分析

针对 2.1 中的每个模块，从三个维度拆解：

- **视觉层 (Visual)**：色彩心理学应用、视觉重心分布、栅格系统利用、字体层级。
- **交互层 (Interaction)**：预期的手势/点击反馈、状态转换逻辑（Loading / Empty / Error / Success）。
- **信息架构 (IA)**：信息的层级权重、字段的组织逻辑、可扩展性。

- **底层规则**：根据 UI 元素推测背后的逻辑限制（库存校验、权限控制、风控规则、算法排序、计费模型等）。

## 质量要求

- **每个推断都要有视觉锚点**：当你说"这里用了红色按钮暗示高风险操作"时，必须能在截图中指出是哪个按钮。避免凭空编造。
- **Mermaid 代码必须语法合法**：使用 `flowchart TD` 或 `sequenceDiagram`，节点 ID 规范、箭头方向清晰、避免嵌套错误。
- **不做廉价结论**："UI 很简洁"、"体验很好"这类空话要删除。洞察必须具体、可落地、可质疑。
- **当截图信息不足时**：显式标注"截图中未呈现，但按同类产品惯例推测……"，不要伪装成观察事实。

## 不适用场景

以下场景请明确告知用户并跳过本 skill：

- 用户只想要纯代码实现，没给截图
- 用户想要文案撰写、翻译、摘要
- 用户想要从图片中 OCR 文本（这是 OCR 任务，不是逆向分析）
- 用户想要绘制新设计稿（这是正向设计任务，方向相反）
