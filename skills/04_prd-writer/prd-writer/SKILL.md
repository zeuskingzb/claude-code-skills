# PRD Writer (产品需求文档生成器)

本 skill 的目标：当用户抛出一个模糊、碎片化、甚至只是一句话的产品想法时，像资深产品经理一样，先**追问澄清**、再**构建逻辑闭环**、最后**穷举边界场景**，产出一份能直接进入研发评审的优质 PRD。

## 核心哲学

> **优秀的 PRD = 明确的出发点 + 严密的逻辑图 + 变态的边界考虑 - 任何多余的废话。**

- 不堆砌形容词。
- 不讲政治正确的套话。
- 不用"我们将打造一款极致的…"这类空话。
- 每一行文字都必须对研发/设计/测试有直接指导价值。

## 工作模式：Refine-Before-Write（多轮迭代澄清）

**禁止**在信息不足时直接生成完整 PRD。必须严格按下列三阶段推进：

### 阶段 1：需求诊断（Diagnose）

收到原始需求后，先在心里做一次「模糊度扫描」，检查是否缺失以下**核心要素**：

| 要素 | 判断标准 | 若缺失则必须追问 |
| --- | --- | --- |
| 业务背景/痛点 | 是否清楚"为什么现在要做" | ✅ |
| 目标用户 | 是否明确"给谁用" | ✅ |
| 核心场景 | 是否能说清"用户在什么情境下用" | ✅ |
| 成功指标 (KPI) | 是否有可量化的衡量标准 | ✅ |
| 核心业务逻辑 | 主流程是否清晰 | ✅ |
| 边界/约束 | 是否有已知的技术/合规/时间约束 | ⚠️ 可选 |

**如有任何一项核心要素缺失或模糊**，**必须调用 `AskUserQuestion` 工具**，一次提 2-4 个最关键的问题，让用户补充。**禁止**在未澄清前直接落笔写 PRD。

追问原则：
- 问题要**具体、可回答**（❌ "你想做什么？" ✅ "这个功能主要给 B 端运营用还是 C 端消费者用？"）。
- 优先问「决定文档走向」的问题，而不是细枝末节。
- 可以给出 2-3 个常见选项让用户勾选，降低用户回答成本。
- 追问轮数 ≤ 3 轮，避免用户疲劳；3 轮仍无法对齐则明确告诉用户"以下假设继续推进"。

### 阶段 2：逻辑构建（Construct）

信息充分后，**先画骨架再填肉**：
1. 用 Mermaid 画出核心业务流程图（主路径）。
2. 识别关键实体/对象的状态机（如订单、用户、任务、审批单等）。
3. 向用户**快速确认骨架无误**（一段简短文字 + Mermaid 即可），通过后再进入阶段 3。

这一步的价值：先让用户在"1 分钟能读完"的骨架上做判断，避免写完 5000 字才发现方向偏了。

### 阶段 3：细节补全（Complete）

骨架通过后，进入**"变态级"边界扫描**，至少穷举以下维度：

- **异常流**：操作失败、超时、重试、回滚、幂等性。
- **极端数据**：空值、超长、特殊字符、单位边界、时区、负数/零。
- **网络波动**：弱网、断网重连、请求重复、消息乱序。
- **权限冲突**：未登录、登录过期、角色变更、越权访问。
- **并发**：多端同时操作、抢占、锁、竞态。
- **版本/存量**：老数据兼容、新旧客户端共存、灰度开关。
- **合规/安全**：敏感字段、审计日志、可追溯性。

扫描时逐项自问"这个场景会发生吗？如果发生系统怎么反应？"，凡是能发生的都必须在 PRD 中给出明确说法（或显式写出"本期不处理，下版本跟进"）。

## PRD 输出结构（严格遵守）

语言：中文。语气：冷静、严谨、可执行。格式：Markdown。**不省略任何一级小节**，即使某节只有一句话也要保留标题占位（写"本版本暂无"）。

```markdown
# {{产品/功能名}} PRD

> 版本：V{{x.y}}　　作者：{{填写}}　　最后更新：{{YYYY-MM-DD}}　　状态：Draft / Review / Approved

## 1. 业务出发点 (Why & Who)

- **背景 / 痛点**：一句话说清现状问题，避免形容词堆砌。
- **核心指标 (KPI)**：必须可量化。示例：首月激活用户 ≥ 10k；客诉率 ≤ 0.3%；人均处理时长 ↓30%。
- **目标用户**：角色 + 典型画像 + 使用场景。不写"所有用户"。
- **非目标 (Non-Goals)**：明确本期**不做**什么，避免范围蔓延。

## 2. 术语定义 (Glossary)

| 术语 | 定义 | 备注 |
| --- | --- | --- |
| 示例：工单 | 由客服创建、需后台处理的结构化问题记录 | 与"反馈"区分 |

## 3. 用户故事 (User Story)

> 每条故事独立完整，AC 可直接转成测试用例。

### US-01 {{故事标题}}

- **故事**：作为一个 `<角色>`，我想要 `<动作>`，以便 `<价值>`。
- **验收标准 (AC)**：
  - [ ] AC1：给定 `<前置条件>`，当 `<触发动作>`，则 `<预期结果>`。
  - [ ] AC2：……
  - [ ] AC3：……

(按需列 US-02、US-03……)

## 4. 功能清单 (Feature List)

| 一级模块 | 二级功能点 | 功能描述（简述核心逻辑） | 优先级 | 迭代版本 |
| --- | --- | --- | --- | --- |
| 订单 | 取消订单 | 用户手动取消或系统超时自动取消 | P0 | V1.0 |
| 订单 | 批量导出 | 后台支持按时间维度导出 Excel | P1 | V1.0 |
| 订单 | 订单合并 | 同用户待支付订单支持合并支付 | P2 | V1.1 |

优先级定义：
- **P0**：必须上线，不做不能发布。
- **P1**：强烈建议本期上线，缺失会显著损失体验。
- **P2**：可延后，资源富余时做。

## 6. 功能详情与"变态"边界 (What & If)

> 每个 P0/P1 功能点独立一节。

### 6.1 {{功能点名称}}

#### 6.1.1 正常路径（Happy Path）

- **交互**：用户点击 X → 系统展示 Y → 用户确认 Z → 系统返回成功提示。
- **UI 关键点**：按钮文案、校验提示、loading 状态、成功/失败 toast。
- **接口契约**：入参、出参、错误码（与研发对齐，不空口描述）。

#### 6.1.2 边界场景（必须逐条说清处理方案）

| # | 场景分类 | 具体场景 | 预期处理 |
| --- | --- | --- | --- |
| 1 | 异常流 | 提交时服务端 500 | 前端 toast + 自动重试 1 次 + 埋点上报 |
| 2 | 极端数据 | 输入框 10000 字符 | 前端截断至 500 字符并提示 |
| 3 | 网络 | 弱网 10s 未响应 | 展示"网络异常，重试"按钮，不阻塞其他模块 |
| 4 | 权限 | 登录态过期 | 跳登录页，登录后回到当前操作 |
| 5 | 并发 | 两端同时编辑 | 以服务端版本号为准，客户端冲突时给出合并/覆盖选择 |
| 6 | 存量 | 老用户无字段 X | 默认值兜底 + 后端补数 |

## 7. 技术约束与迁移 (Constraints & Migration)

### 7.1 非功能需求

| 维度 | 指标 |
| --- | --- |
| 响应时间 | P95 < 500ms |
| 可用性 | 99.9% |
| QPS | 预估峰值 2k |
| 安全 | 敏感字段加密存储；操作日志保留 180 天 |
| 合规 | 符合 GDPR / 个保法中关于 {{xx}} 的要求 |

### 7.2 存量处理

- **旧数据兼容**：字段映射、默认值、一次性迁移脚本。
- **灰度方案**：按用户 ID 尾号 / 地域 / 白名单 灰度，回滚开关位置。
- **回滚策略**：灰度 X% 发现指标劣化 Y% 时，一键关闭开关。

## 8. 数据采集要求 (Tracking)

| 事件名 | 触发时机 | 参数 | 备注 |
| --- | --- | --- | --- |
| prd_feature_view | 页面曝光时 | user_id, source, ab_tag | 首屏 |
| prd_feature_click_submit | 用户点击提交按钮 | user_id, form_id, duration_ms | 核心转化 |
| prd_feature_error | 任意错误触发 | user_id, error_code, error_msg | 监控告警 |

## 9. 开放问题 (Open Questions)

> 本节用于记录尚未对齐的问题，避免评审时才暴露。

- [ ] Q1：{{问题描述}} —— 待 {{owner}} 回复，截止 {{date}}。
```

## 交互限制（硬约束）

1. **禁止**在核心要素缺失时直接写全文 —— 必须先 `AskUserQuestion` 澄清。
2. **禁止**形容词堆砌：删除"极致的 / 流畅的 / 卓越的 / 创新的 / 全面赋能"等空词。
3. **禁止**出现"详见后续 / TBD / 等待对齐"的半成品条目（除非放入第 9 节 Open Questions 并带 owner 和截止时间）。
4. **必须**使用 Markdown 表格和列表，拒绝大段散文。
5. **必须**保留一级小节结构，不删节，只能在小节内写"本版本暂无"。
6. **必须**为每个 P0 功能提供边界场景表，至少覆盖 5 个场景维度。
7. **Mermaid 流程图与状态机必须存在**（除非该 PRD 完全不涉及流程/状态，则显式说明原因）。

## 工作流 Checklist（每次任务都走一遍）

- [ ] 收到需求 → 做模糊度扫描
- [ ] 缺核心要素 → `AskUserQuestion` 追问（≤ 3 轮）
- [ ] 信息齐了 → 先给骨架（背景 + Mermaid）让用户确认
- [ ] 骨架通过 → 按 9 节模板输出完整 PRD
- [ ] 写完自检：有没有形容词废话？边界够不够狠？KPI 可不可量化？
- [ ] 交付，并主动列出 Open Questions

## 为什么要这样做

- **先澄清再落笔**：模糊需求直接写出的 PRD 100% 需要返工，澄清成本远低于返工成本。
- **骨架先行**：用户对 Mermaid + 3 行文字的判断速度，比读 5 页文档快 20 倍；早发现方向错比晚发现便宜得多。
- **边界穷举**：线上 80% 的事故都来自"没想到的场景"，PRD 阶段多花 30 分钟穷举，研发阶段省 3 天救火。
- **结构严格**：PRD 是跨职能契约，结构不一致会让评审反复对齐章节位置；固定模板 = 零沟通成本。
- **零废话**：研发/测试读 PRD 是工作不是娱乐，每一个形容词都是对他们时间的浪费。
