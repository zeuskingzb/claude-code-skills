# 产品需求挖掘 (Product Requirement Mining)

本 skill 的目标：把散落在评论、工单、问卷中的"用户声音"，转化为产品经理可以直接拿去排期的"迭代路线图"。你扮演的角色是**资深产品经理 + 用户研究员 + 数据分析师**三位一体的顾问，四步逐层递进，每一步的产出都是下一步的输入。

## 工作流程总览

| Step | 目标 | 产出 | 文件格式 |
| :--- | :--- | :--- | :--- |
| 1. 关键词分类 | 去噪、要素提取、类型打标 | 每条评论的结构化 JSON | JSON（控制台/中间态） |
| 2. 筛选数据 | 由用户决定聚焦哪一类 | 选定分类的纯文本清单 | `.txt`（一行一条）|
| 3. 聚类分析 | 语义相似项归簇、去重、整合 | 聚类报告 | `.md` |
| 4. 生成路线图 | 优先级评估、验收标准 | 产品优化路线图 | `.md` |

**关键原则**：
- **每一步必须用户确认后再进入下一步**。不要四步连着跑完再给用户看，否则前面一步偏了，后面全部白做。
- **忠于原文**。用户没说的话不要替他说。所有结论都要能回溯到具体评论。
- **文件名带时间戳或产品名**。避免多次运行互相覆盖。

---

## Step 1：关键词分类

### 输入

用户提供的原始评论文本。可能是：粘贴在对话中的多行文本、一个 `.txt` 文件、一个 `.csv` 的某列。先确认数据形态，再开始处理。

### 处理要求

对**每一条**评论，执行以下四步：

1. **去噪清洗**
   - 删除表情符号、无意义语气词（"哈哈哈"、"哎"、"啊啊啊"）、闲聊、重复字符（"好好好好好"→"好"）、纯广告/水军语料。
   - 保留核心业务信息，但不要过度重写——用户的原话语感本身就是信号。

2. **要素提取**
   - **用户信息**：若评论中提及身份/角色/使用场景（"我是一个 5 年老用户"、"做跨境电商的"），提取出来；没提则填 `null`。
   - **痛点描述**：用一句话清晰描述用户遇到的具体问题或不便。
   - **涉及功能模块**：识别该需求归属于产品的哪个模块（如"登录"、"搜索"、"支付"、"消息推送"）。若无法判断，填 `"未识别"`。

3. **自动分类**（四选一，不得自创）
   - **Bug**：报告功能异常/报错/崩溃/数据错误。
   - **建议**：对现有功能的改进想法（调参、优化、体验升级）。
   - **需求**：希望新增的能力/功能/模块。
   - **其他**：纯情绪表达、纯好评、纯无关内容（此时 `is_valid` 一般为 `false`）。

4. **格式输出**（严格 JSON，每条评论对应一个对象）

```json
{
  "is_valid": true,
  "cleaned_text": "清洗后的简洁文本",
  "category": "Bug/建议/需求/其他",
  "module": "涉及的功能模块",
  "user_profile": "用户身份（可为 null）",
  "pain_point": "痛点一句话描述"
}
```

> 为什么要 `is_valid` 字段：有些评论是"好评无信息"或"乱打字符"，标记 `false` 后 Step 2 就可以直接跳过，避免污染后续聚类。

### 输出落盘

将全部条目汇总为一个 JSON 数组，保存到 `step1_classified_<产品名>_<YYYYMMDD>.json`，并向用户展示：
- 总条数
- 四个类别各自的条数
- 发现的主要模块 Top 5

然后**停下来**，等用户确认数据质量 OK 再进入 Step 2。

---

## Step 2：筛选数据

### 交互方式

使用 `AskUserQuestion` 工具询问用户要聚焦哪一类：

- **问题**：本次迭代分析想聚焦哪一类用户声音？
- **选项**（基于 Step 1 实际产出的类别，只展示有内容的）：
  - Bug（共 X 条）
  - 建议（共 Y 条）
  - 需求（共 Z 条）
  - 其他（共 W 条，一般不推荐选）
- **允许多选**：是。用户可能想一次看"Bug + 建议"。

### 输出落盘

把用户选中的类别下所有 `is_valid: true` 的 `cleaned_text`，**一行一条**写入 `step2_filtered_<分类>_<YYYYMMDD>.txt`。

文件头部可加一行注释说明来源和条数，例如：

```
# 来源：step1_classified_XXX.json | 分类：Bug+建议 | 条数：127 | 生成时间：2026-04-21
<评论 1>
<评论 2>
...
```

向用户确认保存路径，进入 Step 3。

---

## Step 3：聚类分析

### 处理要求

读取 Step 2 的 txt 文件，对每条进行聚类。

1. **特征扫描**（对每条数据在心里过一遍，不必全部落盘）
   - **核心意图**：用户到底想解决什么？（如"加快首屏加载"）
   - **关键对象**：围绕哪个功能/页面/数据？（如"订单列表页"）
   - **独特属性**：有没有与众不同的细节？（如"在 iOS 16 上特别慢"）

2. **聚类分组**
   - **相似度阈值**：语义相似度 > 85%，或核心目标一致，即归入同一簇（Cluster）。
   - **识别细微差别**：若两项目标一致但**解决方案不同**（如"想要暗黑模式" vs "想要自定义主题色"），保留为同簇内的"变体"并显式标注。
   - **不要过度合并**：如果两项只是用了相近词但痛点不同，宁可分开。聚类错得太粗，后面路线图就会把不同问题混为一个需求。

3. **去重与整合**
   - **基准项选择**：从每簇中选信息量最全、表达最清晰的一条作为"基准项"。
   - **补全逻辑**：其他重复项若含基准项没有的细节（如具体设备、具体场景、具体数据），把这些细节合并进基准项的描述里。
   - **保留来源条数**：每簇记录去重前的原始条数，这是 Step 4 优先级评估的关键输入（出现越多 = 用户价值越高）。

### 输出格式（严格遵守）

保存为 `step3_clusters_<分类>_<YYYYMMDD>.md`。

```markdown
# 聚类分析报告

- **数据来源**：{step2 文件名}
- **分析日期**：{YYYY-MM-DD}
- **原始条数**：{X}
- **聚类后簇数**：{Y}

---

## 📂 聚类结果

### 1. [聚类主题 A]

- **唯一保留项**：[整合后的精简内容，一句到三句话]
- **去重来源数**：[X] 条
- **涉及模块**：[模块名]
- **变体**（可选）：
  - 方案 a：……
  - 方案 b：……
- **保留理由**：[简述为何此条作为基准项]

### 2. [聚类主题 B]
...
```

### 质量检查

交付前自检：
- 每个基准项是否都能回溯到原始评论？（抽查 3 条试试）
- 是否有过度合并？（一个簇里出现了显然不同的诉求，要拆）
- 是否有过度拆分？（两个簇本质是同一件事，要并）

然后**停下来**，等用户 review 聚类质量。用户可能会让你合并/拆分某些簇——这种反馈非常常见，预留迭代空间。

---

## Step 4：生成产品优化路线图

### 优先级评估框架

对每个簇，从三个维度综合评估：

| 维度 | 评估问题 | 权重参考 |
| :--- | :--- | :--- |
| **用户价值** | 解决后惠及多少用户？痛点多深？是否影响核心路径？ | 去重来源数 × 痛点严重度 |
| **实现成本** | 开发难度高/中/低？是否涉及底层重构？ | 凭经验估算，不确定就标"待评估" |
| **战略对齐** | 是否符合当前阶段目标？是合规/体验/增长哪条线？ | 高/中/低 |

综合映射为四级：

| 级别 | 定义 | 典型特征 |
| :--- | :--- | :--- |
| **P0** | 紧急必做 | 影响核心功能、高频使用、用户严重不满、影响营收或合规 |
| **P1** | 重要优先 | 高用户价值、中等成本、符合战略方向 |
| **P2** | 常规优化 | 中等用户价值、常规改进 |
| **P3** | 远期规划 | 低用户价值、高成本、非当前重点 |

> 为什么要显式给出四级定义：如果直接让你拍"P0 还是 P1"，不同轮的判断会漂移。把锚点写进报告，用户 review 时也有依据挑战你。

### 输出格式（严格遵守）

保存为 `step4_roadmap_<产品名>_<YYYYMMDD>.md`。

```markdown
# {产品名称} 产品优化路线图

## 分析基础

- **数据来源**：{step3 文件名}
- **生成日期**：{YYYY-MM-DD}
- **筛选分类**：{Bug/建议/需求/其他}
- **去重后有效需求**：{X} 条
- **聚类簇数**：{Y} 个

## 优先级定义

| 级别 | 定义 | 典型特征 |
| :--- | :--- | :--- |
| **P0** | 紧急必做 | 影响核心功能、高频使用、用户严重不满、影响业务 |
| **P1** | 重要优先 | 高用户价值、中等成本、符合战略方向 |
| **P2** | 常规优化 | 中等用户价值、常规改进 |
| **P3** | 远期规划 | 低用户价值、高成本、非当前重点 |

---

## 优化路线总览

| 优化项 ID | 优化主题 | 需求来源 | 需求类型 | 优先级 | 负责模块 | 预期目标 | 验收标准 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| {模块}-Bug-01 | {解决 XXX 问题} | {聚类主题}，来源 {X} 条评论 | Bug | P0 | {模块名} | {改进目标} | {验收条件} |
| {模块}-建议-01 | {新增 XXX 能力} | {聚类主题}，来源 {X} 条评论 | 建议 | P1 | {模块名} | {改进目标} | {验收条件} |

---

## 详细问题说明

### {模块}-Bug-01：{优化主题}

**优先级**：{P0/P1/P2/P3}  |  **模块**：{模块名}  |  **用户反馈**：{X} 条

**问题描述**

{对聚类基准项的完整复述，保留用户原始语境}

**主要症状**

- {症状 1}
- {症状 2}
- {症状 3}

**用户影响**

{说明这个问题对用户体验/业务指标的具体影响}

**预期目标**：{一句话的改进目标，可度量}

**验收标准**：{可验证的验收条件，最好能对应到测试用例或指标口径}

---

（对每个聚类簇重复以上结构）
```

### 质量要求

- **ID 必须可追溯**：`{模块}-{类型}-{序号}` 能让产品和研发在后续评审里快速定位。
- **验收标准要可度量**："加载变快"不算；"首屏 TTFB < 800ms（P75）"才算。如果没法量化，至少要写"可复现的操作步骤下，现象消失"。
- **不要 P0 满天飞**：如果超过 30% 的簇都被标 P0，你就是在逃避决策。回头重新掂量。
- **战略对齐要说人话**：不要写"符合公司愿景"这种空话，写"配合 Q2 XX 功能上线后的体验闭环"这种具体的。

---

## 四步收尾

全部完成后，给用户一个一屏可见的总结：

```
✅ 产品需求挖掘完成
├─ Step 1 分类：<文件路径>（共 X 条，Bug Y / 建议 Z / 需求 W / 其他 V）
├─ Step 2 筛选：<文件路径>（聚焦 <分类>，X 条）
├─ Step 3 聚类：<文件路径>（Y 个簇）
└─ Step 4 路线图：<文件路径>（P0 a / P1 b / P2 c / P3 d）
```

然后问：**是否需要我基于 P0/P1 生成首批排期建议，或换一个分类（如"建议"）再跑一轮？**

---

## 不适用场景

以下情况请明确告知用户并跳过本 skill：

- 用户只想做**情感极性打分**（正/负/中）或 NPS 统计——那是情感分析任务。
- 用户想做**竞品对比**而非自家产品挖掘——那是竞品分析任务。
- 没有任何原始评论文本，用户想让你"凭空想三个需求"——那是需求脑暴，不是挖掘。
- 评论量极少（< 10 条）——聚类没统计意义，直接人工过一遍更快，建议用户先攒数据。
