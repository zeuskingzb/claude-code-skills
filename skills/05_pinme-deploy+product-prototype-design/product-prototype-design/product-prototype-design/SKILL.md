# 产品原型设计 (Product Prototype Design)

本 skill 的目标：当用户抛出一个产品想法、PRD 片段或改版诉求时，像资深产品经理 + 高级交互设计师 + 前端工程师三位一体地产出一个**能点、能跳转、能看到数据、能演示**的高保真单 HTML 原型，并一键部署到可访问链接。

## 核心哲学

> **好原型 = 对的行业灵魂 + 对的视觉规范 + 能跑的 Mock 数据 + 一键可分享。**

- 不做"截图式"静态稿——原型必须能交互、能点击、能看到状态切换。
- 不做"过度工程"——不引入打包器、不拆分多文件、不搞框架脚手架。
- 不做"方向性赌博"——行业推断不确定时必须先问，不要押错色系。
- 不做"假数据孤岛"——Mock 数据必须贴合业务语境（金融就是金额/账户，电商就是 SKU/订单），不用 Lorem Ipsum。
- 不做"半成品交付"——写完必须部署，回传**可访问链接**给用户。

## 工作流程（强制 4 步，不得跳步）

### Step 1：推断行业 / 场景

收到需求后，先做一次**行业画像扫描**，根据用户提供的背景（产品描述、PRD、目标用户、关键词）对照下表做归类：

| 行业/场景 | 典型关键词信号 |
| :--- | :--- |
| 科技/互联网 | SaaS、API、开发者、AI、云、平台、Dashboard |
| 金融/银行 | 银行、理财、账户、交易、基金、保险、合规、KYC |
| 电商/零售 | 商品、SKU、订单、购物车、促销、秒杀、店铺 |
| 餐饮/美食 | 餐厅、菜单、外卖、点餐、食谱、预订 |
| 医疗/健康 | 医院、挂号、问诊、病历、健康、药品、医保 |
| 教育/培训 | 课程、学员、讲师、作业、考试、培训、学习 |
| 美妆/时尚 | 美妆、护肤、穿搭、品牌、新品、种草 |
| 房地产/家居 | 房源、楼盘、户型、家具、装修、物业 |
| 游戏/娱乐 | 游戏、玩家、对战、排行榜、角色、皮肤、直播 |
| 汽车/工业 | 车型、试驾、4S、生产、产线、工厂、制造 |
| 法律/咨询 | 律师、合同、案件、合规、咨询、律所 |
| 旅游/出行 | 机票、酒店、行程、旅游、景点、打车、地图 |
| 公益/环保 | 公益、捐赠、环保、志愿者、碳、绿色 |
| 企业官网（通用） | 公司介绍、Solution、About、Contact、品牌官网 |

**判定规则：**
- **信号明确**（至少 2 个关键词命中）→ 直接锁定行业，进入 Step 2。
- **信号冲突/不足**（无法在上表 1 分钟内归类）→ **必须主动向用户追问**，一次问清楚：
  - "这个产品主要服务的行业是？（可给 2-3 个候选让用户勾选）"
  - "主要使用场景是 Web 后台 / 移动 App / 官网 Landing Page / 其他？"
  - "目标用户画像有没有偏好（比如年轻潮流 vs 严肃专业）？"
- **禁止**在行业未定的情况下凭直觉开工。

### Step 2：选择视觉设计规范（强制对照执行）

**2.1 配色方案**：按 Step 1 锁定的行业，严格从下表取主色/辅助色/中性色。不得自创、不得跨行业混搭。

| 行业/场景 | 主色调 | 辅助色 | 中性色 | 风格特点 |
| :--- | :--- | :--- | :--- | :--- |
| 科技/互联网 | 蓝色（深蓝/宝蓝） | 浅蓝、青、白 | 白、浅灰、深灰 | 专业、冷静、科技感、信任 |
| 金融/银行 | 藏蓝、墨绿、深红 | 金、浅灰 | 白、浅灰 | 稳重、安全、权威、高端 |
| 电商/零售 | 红、橙、橘黄 | 黄、浅蓝、白 | 白、浅灰 | 活泼、促销感、醒目、促转化 |
| 餐饮/美食 | 暖橙、暖红、暖黄 | 浅绿、米黄 | 米白、浅棕 | 食欲、温暖、烟火气、清新 |
| 医疗/健康 | 浅绿、浅蓝、白色 | 淡粉、浅灰 | 白、浅灰 | 干净、安心、专业、柔和 |
| 教育/培训 | 蓝色、绿色 | 黄、浅橙 | 白、浅灰 | 理性、成长、清爽、易读 |
| 美妆/时尚 | 粉、紫、香槟金 | 浅灰、白、银 | 白、浅灰、黑 | 精致、轻奢、高级、女性化 |
| 房地产/家居 | 咖色、深棕、墨绿 | 米白、浅金 | 白、米灰 | 质感、稳重、高端、温馨 |
| 游戏/娱乐 | 紫、蓝、霓虹色 | 亮粉、亮黄、青 | 黑、深灰 | 潮流、炫酷、年轻、视觉冲击 |
| 汽车/工业 | 黑、深灰、深蓝 | 红、银、白 | 白、浅灰 | 硬朗、科技、高级、力量感 |
| 法律/咨询 | 藏蓝、深灰、黑 | 金、暗红 | 白、浅灰 | 严谨、正式、专业、可信赖 |
| 旅游/出行 | 天蓝、湖蓝、草绿 | 白、浅橙 | 白、浅灰 | 开阔、清爽、自然、放松 |
| 公益/环保 | 绿色、浅蓝 | 白、浅黄 | 白、浅灰 | 自然、希望、清新、正能量 |
| 企业官网（通用） | 深蓝/深绿 | 浅蓝、白 | 白、灰、黑 | 大气、正式、品牌感强 |

**2.2 视觉风格**：按行业注入"灵魂"，字体 / 容器 / 交互 / 布局必须一起切换，不得只改颜色。

| 行业/场景 | 字体排版 (Typography) | 容器与边角 (Radius/Shadow) | 交互动效 (Interaction) | 布局与空间 (Layout) |
| --- | --- | --- | --- | --- |
| 科技/互联网 | `font-sans`, 字母间距收紧 `tracking-tight` | 极细边框 `border-slate-100`, 低阴影 | 悬停平滑位移 `hover:-translate-y-1` | 强调负白空间，使用 `gap-8` 以上大间距 |
| 金融/银行 | 标题用 `font-serif` 或粗体，体现权威 | 直角或小圆角 `rounded-sm`, 投影沉重 `shadow-md` | 严谨加载动画，避免花哨弹跳 | 模块化网格，左右对齐极度严谨 |
| 电商/零售 | 价格文字放大并加粗，使用醒目色彩 | 大圆角 `rounded-2xl`, 强阴影 `shadow-xl` | 按钮呼吸灯 `animate-pulse` | 瀑布流布局，CTA 始终悬浮 |
| 餐饮/美食 | 手写体或圆润字体，增加亲和力 | 有机形状（不规则圆角），背景 `bg-orange-50/30` | 图片悬停放大 `hover:scale-105` | 大图驱动，卡片式布局，突出食材质感 |
| 医疗/健康 | 增加行高 `leading-relaxed`，提升易读性 | 中等圆角 `rounded-lg`, 柔和阴影 `shadow-blue-100/50` | 淡入淡出 `transition-opacity` | 极简主义，去除所有干扰装饰 |
| 教育/培训 | 清爽无衬线体，重点内容加背景高亮 | 模块边框 `border-2`, 活泼彩色阴影 | 进度条动画，点击反馈明显 | 线性叙事，分步式引导 (Step-by-step) |
| 美妆/时尚 | 标题超大且字间距拉开 `tracking-widest` | 极细线条，去阴影或仅留背景色块切换 | 优雅淡入动画，滚动视差效果 | 杂志排版风格，大量非对称留白 |
| 房地产/家居 | 标题采用现代衬线体，显得贵气 | 容器使用内阴影 `shadow-inner` 增加质感 | 平滑缩放切换 | 宽屏大图，强调空间延伸感 |
| 游戏/娱乐 | **强制深色模式**，标题倾斜 `italic` | 霓虹发光边框 `ring-1 ring-cyan-400`（**禁渐变**） | 剧烈反馈，如 `active:scale-95` | 碎片化、重叠式布局，打破常规栅格 |
| 汽车/工业 | 全大写标题 `uppercase`, 高对比度 | 锐利边角，金属质感边框 `border-slate-700` | 极速切换，硬朗线性滑动 | 强调性能参数对比表，暗色基调 |
| 法律/咨询 | 经典宋体/衬线体标题，正文严谨 | 对称布局，严禁花哨装饰 | 极其克制的动效，仅限必要导航 | 经典 F 型阅读路径，强调文档和资质展示 |
| 旅游/出行 | 字体轻盈，标题带轻微投影 | 图片圆角 `rounded-3xl`，卡片悬浮感 | 流畅滚动加载 | 宽大视野，背景图穿插，模拟开阔感 |
| 公益/环保 | 柔和字体，大量使用图标辅助 | 叶子形状圆角 `rounded-tl-3xl rounded-br-3xl` | 温暖渐显效果 | 自然流畅的 S 型布局，多用矢量插画 |
| 企业官网 | 均衡比例，主标题稳重 | 标准圆角 `rounded-md`, 规范化阴影 | 稳定平滑滚动 | 传统 Hero + Features + Solution 结构 |

**2.3 禁忌约束（不得违反）**：

- ❌ **严禁**使用任何形式的"蓝色到紫色"渐变色（No Blue-Purple Gradients）。
- ❌ **严禁**使用 `from-blue-500 to-purple-500` / `from-indigo-500 to-purple-500` / `bg-gradient-to-r from-blue-* to-purple-*` 等 Tailwind 组合。
- ❌ 不得跨行业混用色系（例如给银行产品用电商橘红）。
- ❌ 不得偏离 2.2 表格里的风格锚点（例如给游戏/娱乐用白底浅色）。

### Step 3：创建原型设计

**3.1 核心目标**：构建一个**高保真、可点击、含 Mock 数据**的单 HTML 交互原型，验证 UI 布局与用户体验。

**3.2 技术栈（严格固定，不得替换）**：

- HTML5
- TailwindCSS（Utility-First，CDN 引入）
- Vanilla JavaScript (ES6+)
- Lucide Icons（图标库，CDN 引入）
- Faker.js 或 Mock.js（Mock 数据生成，CDN 引入）

**3.3 CDN 引用模板**（放入 `<head>`）：

```html
<script src="https://cdn.tailwindcss.com"></script>
<script src="https://unpkg.com/lucide@latest"></script>
<!-- 二选一：英文语境用 Faker.js，中文语境优先用 Mock.js -->
<script src="https://cdn.jsdelivr.net/npm/@faker-js/faker@8.4.1/dist/index.umd.min.js"></script>
<script src="https://cdn.jsdelivr.net/npm/mockjs@1.1.0/dist/mock.min.js"></script>
```

**3.4 原型必备要素**（缺一不可）：

1. **完整页面骨架**：至少包含导航栏 / 主内容区 / 次要信息区 / 底部/状态区，不得只有一个孤立模块。
2. **真实业务 Mock 数据**：列表、表格、卡片不得少于 5 条数据，数据语义必须贴合行业（金融就写"招商银行储蓄卡 **** 8821"，不要写"Item 1"）。
3. **可交互状态**：至少实现以下任意 3 项——
   - 导航/Tab 切换
   - 弹窗/抽屉的打开关闭
   - 搜索/筛选（前端过滤 Mock 数据）
   - 表单输入 + 提交反馈（不走真实接口，弹 Toast/Modal 即可）
   - Loading / Empty / Error / Success 四态展示
4. **响应式布局**：至少在 `md` 断点做一次栅格重排，移动端原型必须做 `max-w-[480px] mx-auto` 的手机画布包裹。
5. **图标与插图**：用 Lucide 图标注入细节，不得用文字代替图标。
6. **配色与风格一致性**：严格套用 Step 2 锁定的配色 + 风格表，不得"灵光一现"临时改色。

**3.5 代码结构建议**：

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />
  <title>[产品名] - 原型</title>
  <!-- CDN: tailwind, lucide, mockjs/faker -->
  <style>
    /* 仅放少量 Tailwind 覆盖不了的自定义样式，如滚动条/动画关键帧 */
  </style>
</head>
<body class="[行业对应的底色/字体类]">
  <!-- 1. 导航/顶部栏 -->
  <!-- 2. 主内容区（按行业布局规则组织） -->
  <!-- 3. 侧栏/辅助信息 -->
  <!-- 4. 底部/状态 -->

  <script>
    // 1. 生成 Mock 数据
    // 2. 渲染列表/卡片
    // 3. 绑定交互事件（Tab、弹窗、筛选、表单等）
    // 4. lucide.createIcons();
  </script>
</body>
</html>
```

**3.6 交付物**：一个独立可运行的 `.html` 文件。文件命名建议：`[产品名]-prototype.html`，保存至当前工作目录或用户指定目录。

**3.7 质量自检清单（生成后必查）**：

- [ ] 直接双击 `.html` 文件能在浏览器打开，无 404 / 无控制台报错。
- [ ] 颜色、字体、圆角、阴影均与 Step 2 表格对齐（可逐项对照）。
- [ ] 没有出现"蓝→紫"渐变类名。
- [ ] Mock 数据不少于 5 条，且语义贴合行业。
- [ ] 至少 3 种可交互状态工作正常。
- [ ] 所有图标都通过 `lucide.createIcons()` 成功渲染（没有空白占位）。
- [ ] 移动端场景下已套 `max-w-[480px]` 手机画布。

### Step 4：网站部署

原型生成并通过 3.7 自检后，**必须**立即调用 `pinme-deploy` skill 完成部署，不得停留在"交付 HTML 文件"这一步。

**执行方式**：

1. 告知用户："原型已生成，接下来发布到 IPFS 以便分享预览。"
2. 调用 `pinme-deploy` skill：因原型为纯 HTML 单文件，属于表中"纯 HTML"项目类型——**无需构建**，直接在 HTML 文件所在目录执行 `pinme upload .` 即可。
3. 将 pinme 返回的 **CID / 访问链接** 显眼地回传给用户。
4. 附带提示：`pinme list` 可查历史、`pinme rm <hash>` 可撤回。

**部署注意事项**：

- 若目录里除了 HTML 还有无关文件（如 node_modules/、.git/、源码），**必须**先把 HTML 单独放到一个干净子目录再 upload，避免把敏感/体积大的文件一起发布。
- 若 HTML 内部使用了 history 路由（虽然原型一般不会）——**改为 hash 路由**再部署，否则刷新会 404。

## 为什么要这样做

- **行业先判**：配色/排版/交互都是服务于"信任感"的，银行用电商橘红会瞬间失去严肃感——所以 Step 1 必须先锁定行业再动手。
- **视觉规范强约束**：原型质量的 80% 来自"不出错的配色+字体+间距"，而不是花哨的动效，所以 Step 2 的表格是硬约束不是建议。
- **单 HTML + CDN**：工程化原型（Next.js/Vite）跑起来要 1 分钟、改一行刷一次、还要部署构建——**原型的价值是"快到能迭代"**，单 HTML 双击即跑、一键发布是最优解。
- **Mock 数据贴合业务**：评审会/演示现场，"招商银行储蓄卡 **** 8821" 的说服力远大于 "Item 1"——原型要让 stakeholder 秒懂业务闭环。
- **强制部署**：设计稿/HTML 文件不能 WhatsApp/微信直接甩给老板，必须是一个链接——所以 Step 4 不是可选项。

## 常见错误与处置

| 错误现象 | 根因 | 修正方式 |
| :--- | :--- | :--- |
| 配色像"通用模板"，没有行业辨识度 | 跳过了 Step 2，直接默认蓝灰 | 回到 Step 1 确认行业，严格查 Step 2 表格逐项替换 |
| 出现 `from-blue-500 to-purple-500` 类渐变 | 忽视 2.3 禁忌约束 | 全局搜索 `from-*-* to-*-*` 和 `bg-gradient-*`，改为纯色块 + 边框 |
| Mock 数据是 "Item 1, Item 2" | 未绑定业务语境 | 用 Mock.js/Faker 生成符合行业的字段（金额、订单号、病历号、商品名） |
| 图标位置有空白方块 | 忘记调用 `lucide.createIcons()` | 在所有 DOM 渲染完成后执行一次 `lucide.createIcons()` |
| 移动端原型铺满整个桌面屏幕 | 没套手机画布 | 用 `<div class="max-w-[480px] mx-auto min-h-screen shadow-2xl">` 包裹 |
| 部署后刷新子页面 404 | 用了 history 路由 | 改为 hash 路由或用锚点 `#/xxx` |
| 部署后访问是源码目录 | 上传目录里缺 `index.html` | 把原型 HTML 重命名为 `index.html` 后再 `pinme upload` |

## 与用户的交互风格

- 每一步开始前先**说一句你要做什么**（"先定行业 → 再定视觉规范 → 再生成代码 → 最后部署"），再执行。
- 行业/场景不明确时，**一次问清楚** 2-4 个关键问题（行业、平台形态 Web/Mobile/Landing、目标用户气质、核心功能模块），不要挤牙膏式追问。
- 生成代码时**先说明**将使用的主色/风格锚点（例如："按 Step 2 锁定: 科技/互联网 → 宝蓝 + 浅灰 + `tracking-tight` + `hover:-translate-y-1`"），再落笔，便于用户中途纠偏。
- 部署成功后，把**访问链接**单独一行醒目贴出，并附带"如需调整样式/内容，直接告诉我需要改什么"的邀请。

## 不适用场景

以下场景请明确告知用户并**跳过本 skill**（或建议其他路径）：

- 需要真实登录态、真实数据库、真实支付回调的全栈实现 → 建议进入工程化开发。
- 需要多页 SPA 路由 + 状态管理 + 构建工具 → 推荐 Vite/Next.js 工程化脚手架。
- 只要一张视觉稿（PNG/Figma）而不要代码 → 转用纯设计工具。
- 只要文案/文档/PRD → 使用 `prd-writer` skill。
- 只要从已有截图逆向分析产品 → 使用 `product-reverse-analysis` skill。
